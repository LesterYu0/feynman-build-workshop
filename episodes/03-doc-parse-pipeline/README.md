## 你将面对什么（背景）

私募尽调项目，每天 200+ 文档（招股书 / 审计报告 / 行业研究 / Excel 股权结构 / Word 合同）：

| 文档类型 | 占比 | 关键诉求 |
|---|---|---|
| 招股书数字 PDF | 40% | 抽表格（财务报表） |
| 审计报告扫描件 | 30% | OCR 识别（数字 PDF 抽不到文字）|
| 行业研究报告 | 20% | 抽文字 + 表格（混合版式）|
| Excel 股权结构 | 5% | 多 sheet + 合并单元格 + 公式 |
| Word 合同 | 5% | 段落 + 表格 + 标题层级 |

**核心矛盾**：每种文档的最优解不同，但业务方要的是**统一的结构**（Document + Chunk），且**业务字段必须全命中**（少一个字段就得退回）。

## 7 层 Pipeline 架构

```
┌─────────────────────────────────────────────┐
│ L7 业务验收：关键字命中 + 行号 + chunk数     │
├─────────────────────────────────────────────┤
│ L2 Parser 路由：按文件类型 → 7路 parser     │
├─────────────────────────────────────────────┤
│ L3 PDF 4 档    →  L4 OCR 4 降级             │
│   档1 PDF-dots       L1 Tesseract（英文）    │
│   档2 PDFPlumber     L2 PaddleOCR（中文）    │
│   档3 PyMuPDF        L3 Surya（旋转/复杂）  │
│   档4 MinerU         L4 VLM（终极方案）     │
├─────────────────────────────────────────────┤
│ L5 Excel 多表：openpyxl 拆 sheet + 公式     │
│ L6 Word 统一  ：python-docx 段+表+图        │
├─────────────────────────────────────────────┤
│ L1 Schema 定义 ：统一 Document / Chunk      │
└─────────────────────────────────────────────┘
```

**为什么是 7 层**（不是 3 层或 10 层）：
- **L1 Schema** 是地基，没它业务方拿不到统一结构
- **L2 路由** 是入口，没它业务方得自己判断走哪
- **L3 + L4** 是 PDF 核心：4 档对应 4 个时延/能力档位，OCR 4 降级是档 4 的兜底
- **L5 + L6** 是 Excel/Word 专用：每种文件有独立坑，必须独立处理
- **L7 验收** 是底线：没它你不知道解析对了没有

**4 档 vs 4 降级**（很多同学会混）：
- **4 档 = L3 PDF 4 档**：选哪一档解析策略
- **4 降级 = L4 OCR 4 级**：扫描件 OCR 时，逐级升档

## 5 大取舍

### 取舍 1：解析速度 vs 表格完整性
- **快**：档 1（pypdf）只抽文字，0.5s/页
- **全**：档 2（pdfplumber）抽表格，1-2s/页
- **平衡**：90% 情况用档 2（招股书必须有表格）

### 取舍 2：免费 vs 复杂版式
- **免费**：档 1-3 全免费（PyMuPDF 已很强大）
- **复杂**：档 4（MinerU）需要 GPU 部署
- **平衡**：招股书/审计报告数字版走档 2-3，扫描件才上档 4

### 取舍 3：抽文本 vs 抽结构
- **文本**：pypdf 简单抽，但表格错位
- **结构**：PyMuPDF 保留 bbox + DataFrame
- **平衡**：要引用原文行号 → PyMuPDF 档 3

### 取舍 4：单个 parser vs 多 parser 路由
- **单**：pdfplumber 处理所有 PDF
- **多**：按文件类型 + 复杂度路由 7 路 parser
- **平衡**：多 parser 路由（每天 200 文档，单 parser 撑不住）

### 取舍 5：业务验收内置 vs 外置
- **外置**：解析完再写验收脚本
- **内置**：`doc.validate(keywords)` 一行调用
- **平衡**：内置（每天跑 200 次，外置会忘）

## 4 大硬骨头（实战翻车现场）

### 硬骨头 1：OCR ≠ 文档解析
- 拍脑袋想：扫描件 PDF → 调用 OCR API → 拿到文字
- **翻车**：OCR 能识别"这是发票"，但**不知道哪是金额、哪是日期、哪是发票号**
- **修复**：OCR 只是 L4 一档，必须和 L3（PDF 结构）+ L7（业务验收）配合

### 硬骨头 2：Excel 合并单元格
- 拍脑袋想：openpyxl 读所有单元格
- **翻车**：合并单元格"股东持股 30%"，只写在第一格，其他格子是 None
- **修复**：遍历合并区，**给每个被合并的格子都填值**

### 硬骨头 3：扫描件旋转 90°
- 拍脑袋想：MinerU 调一下
- **翻车**：旋转 30° 的扫描件，OCR 识别率从 95% 降到 30%
- **修复**：4 级降级（先 Tesseract，再 PaddleOCR，再 Surya，最后 VLM 兜底）

### 硬骨头 4：模糊 PDF（低分辨率扫描）
- 拍脑袋想：调高 OCR DPI
- **翻车**：DPI 调高后时延从 10s/页涨到 60s/页，业务方等不了
- **修复**：先做**业务关键字预检**，如果 60% 关键字都没命中，直接走 L4 档 4（VLM）

## 关键决策速查

### 5 分钟判断走哪档

| 文档类型 | 数字 PDF | 扫描件 |
|---|---|---|
| 5-10 页快速预览 | 档 1 | 档 4 |
| 数字 PDF + 表格 | **档 2** ✅ | 档 4 |
| 复杂排版（多栏） | 档 3 | 档 4 |
| 1000+ 页批量 | 档 2 | 档 4 + 抽样 |
| 终极保底 | 档 4 | 档 4 |

> **90% 情况下 → 数字 PDF 走档 2，扫描件走档 4**

### 何时升级档位

| 信号 | 升级到 |
|---|---|
| 业务关键字命中 < 80% | 档 2 → 档 3 |
| 表格行号缺失 | 档 2 → 档 3 |
| 扫描件 OCR 置信度 < 0.6 | 档 4 + L4-3 Surya |
| 旋转 30°+ | 直接 L4-3 Surya |
| 数学公式识别 | 直接 L4-4 VLM |

## 实战经验总结

### 3 个黄金法则
1. **数字 PDF 90% → 档 2**：不要无脑上档 4（慢 5-10 倍）
2. **扫描件直接档 4**：档 1-3 对扫描件都是 0 字符输出
3. **业务验收必跑**：解析后必跑 `doc.validate(keywords)`，否则你不知道解析对了没有

### 2 个常见翻车
- ❌ **无脑走档 4**：所有 PDF 都走档 4，慢 5-10 倍，90% 情况浪费
- ❌ **跳过业务验收**：解析完直接入库，解析错了也不知道

### 1 个血泪教训
- OCR 识别率不是越高越好，**业务字段命中数才是金标准**
- 一份扫描件 OCR 置信度 0.95，但"总资产"被识别成"总资立" → 业务验收失败
