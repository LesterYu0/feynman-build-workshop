# 3层选型决策树

## 快速路径

### 你的文档有领域本体吗？

**有 → 强制 Ontology RAG (Layer 3)**

- 已有 OWL Schema 或领域分类体系？
  - ✅ 直接导入，LLM 按 schema 抽取实体和关系
  - ❌ 没有？用 OntoRAG 自动归纳：聚类→社区发现→属性聚合→子社区精炼→多层级本体
- 核心收益：消灭30% LLM抽取幻觉 + 93.7% 表格F1
- 什么时候一定用？
  - 合规（条款覆盖判断）
  - 法律（法条交叉引用）
  - 工业标准（ASTM/API/ISO解析）
  - 医疗（SNOMED/ICD分类）

**没有 → 你的答案需要跨chunk推理吗？**

**需要纵向概览→细节 → RAPTOR (Layer 2a)**

- 特征：问题涉及"全文主题""层次结构""从宏观到微观"
- 典型问题："这份文档的核心论点是什么""从顶层设计到底层实现怎么衔接"
- 构建方式：
  1. 叶子 = 原始 chunk
  2. GMM 软聚类（一个 chunk 可属多个簇）
  3. LLM 对每个簇生成摘要 = 上层节点
  4. 递归直到根节点
- 查询：树遍历（深度优先选最相关子节点） 或 折叠树（所有节点平铺并行比选）

**需要横向实体关系 → GraphRAG (Layer 2b)**

- 特征：问题涉及"谁和谁有什么关系""哪些实体依赖这个"
- 典型问题："哪个供应商依赖这个组件""哪条条款覆盖了这个案例"
- 构建方式：
  1. LLM 从 chunk 中抽取实体和关系
  2. 构建知识图谱
  3. Leiden 聚类 → 社区
  4. LLM 为每个社区写摘要报告
- 查询：
  - Local Search：实体邻居遍历 → 组装子图
  - Global Search：社区摘要 map-reduce
- ⚠️ 大坑：30% 是幻觉 → 如有本体，升级到 Layer 3

**不需要跨chunk → 标准 RAG (Layer 1)**

| 业务场景 | 推荐策略 | chunk_size | 理由 |
|---------|---------|-----------|------|
| 通用QA/FAQ | Recursive | 400-512 | 零额外成本，2026通用冠军 |
| 长文档强引用（合同/手册） | Late Chunking | 600-800 | 保留跨句引用信号 |
| 召回要求极高（法律/医疗） | Contextual+RRF+Rerank | 512 | 91% recall，成本30x |
| 聊天记录/会议转录 | Fixed-size | 512 | 格式统一，最简单 |
| 叙事文本/小说段落 | Semantic | 400-600 | 主题边界切割（注意碎化风险） |
